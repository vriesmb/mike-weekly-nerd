Testen als een echte gebruiker – Clarke Verdel

Hoe bouw je componenten die écht betrouwbaar zijn? Clarke Verdel liet zien hoe hij dit doet bij Miele, waar hij werkt aan internationale e-commerce projecten als T-shaped engineer.

★ ★ ★ ★ ★

Onderwerpen
Clarke werkt aan MieleAutocomplete: een eigen design system met herbruikbare componenten. De uitdaging is om die componenten betrouwbaar te maken voor meerdere landen, browsers en gebruikers. Denk aan dark mode, performance, SEO en toegankelijkheid (A11y).

Hij werkt volgens de Testing Trophy van Kent C. Dodds: een piramide met de volgende lagen:
  • Static Tooling (linting, type checking)
  • Unit Tests
  • Integration Tests
  • End-to-End (E2E) Tests

Tools & Stack
  • ESLint – Voor codekwaliteit
  • TypeScript – Type checking
  • Vitest – Unit & Component testing
  • Jest – Voor snelle tests in synthetic DOM
  • Cypress / Playwright – E2E tests in de browser
Clarke benadrukte dat browsergebaseerd testen (met bijvoorbeeld Playwright) visuele checks en debugging mogelijk maakt, maar ook trager is dan lokale tests.

Voorbeelden & Uitleg
Een component wordt eerst lokaal getest met unit- en component tests. Denk aan: werkt de knop correct? Rendeert de juiste tekst?

Daarna volgt integratie: werkt het geheel samen als verwacht? Bijvoorbeeld: klikken op een knop opent een modal met correcte data.

De zwaarste testvorm is E2E: deze bootst echte gebruikersinteractie na. De tests draaien in de browser en gebruiken echte data. Denk aan: een gebruiker logt in, zoekt een product en rekent af.

Ideeën & Toepassing
Clarke ziet browser testing steeds belangrijker worden. Dankzij tools als Vitest kun je tegenwoordig zelfs componenten direct in de browser testen, wat debugging vergemakkelijkt.

Ook het meenemen van accessibility checks in de browser is eenvoudiger: je ziet direct wat werkt en wat niet. Daardoor is de kans groter dat je écht bouwt wat de eindgebruiker nodig heeft.

Reflectie
Wat ik meeneem van Clarke’s verhaal is het belang van vertrouwen. Vertrouwen dat je code doet wat je denkt dat het doet – en dat krijg je alleen als je test vanuit het perspectief van de gebruiker.

Clarke liet zien dat E2E tests onmisbaar zijn voor de complexiteit van moderne webapplicaties. Zeker als je, net als bij Miele, te maken hebt met tientallen landen en duizenden gebruikersscenario’s.

Het deed me ook inzien dat een goede setup (met type checking, linters, en component testing) niet alleen bugs voorkomt, maar ook ontwikkelsnelheid verhoogt. Het is niet alleen veiliger bouwen, maar ook slimmer.
